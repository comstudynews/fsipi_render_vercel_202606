# 개발 일정

## 1. 일정 기준

- 대상: 미래스마트산업진흥원 홈페이지 1차 MVP
- 방식: 바이브코딩과 수동 검토 병행
- 권장 기간: 6주
- 전제: 기관 기본정보와 초기 콘텐츠 담당자가 별도로 준비
- 개발 인원: 1명 중심, 필요 시 디자인·콘텐츠 검수 지원

바이브코딩은 구현 속도를 높일 수 있지만 요구사항, 데이터 구조, 보안, 개인정보, 테스트는 사람이 직접 검토해야 한다.

## 2. 전체 일정

| 주차 | 단계 | 주요 작업 | 핵심 산출물 |
|---:|---|---|---|
| 1주차 | 기획 확정 | 목적, 사용자, 사이트맵, 기능, 콘텐츠 범위 확정 | 기획서 9종, 콘텐츠 목록 |
| 2주차 | UX·디자인 | 와이어프레임, 디자인 시스템, 반응형 기준 | 화면설계서, UI 컴포넌트 |
| 3주차 | 프런트 기반 | 프로젝트 생성, 공통 레이아웃, 공개 정적 화면 | 메인, 소개, 사업 화면 |
| 4주차 | 백엔드·관리자 | DB, API, 인증, 콘텐츠 CRUD, 파일 업로드 | 관리자 MVP, 게시판 API |
| 5주차 | 신청·문의 연계 | 교육, 신청, 문의, 이메일, 검색 | 신청·문의 흐름 |

## 3. 주차별 상세 일정

### 3.1 1주차: 기획 확정

① 참고 사이트 분석 결과 검토  
② 기관의 공식 정보 확인  
③ 1차 메뉴와 제외 범위 확정  
④ 사용자와 핵심 과업 정의  
⑤ 콘텐츠 담당자와 초기 자료 목록 확정  
⑥ 개인정보 수집 항목과 처리방침 검토

**완료 기준**

- 상위 메뉴 6개 확정
- 주요사업 4~6개 확정
- MVP 기능 목록 승인
- 공식 연락처와 기관정보 확보 여부 확인

### 3.2 2주차: UX·디자인

① 메인 페이지 와이어프레임  
② 서브페이지 공통 레이아웃  
③ 게시판 목록·상세  
④ 교육 상세·신청 폼  
⑤ 관리자 화면 구조  
⑥ 모바일 화면 검토  
⑦ 로고, 색상, 폰트, 버튼 규칙 정의

**완료 기준**

- 핵심 화면의 데스크톱·모바일 와이어프레임
- 디자인 토큰 확정
- 화면별 입력·오류·빈 상태 정의

### 3.3 3주차: 프런트엔드 기반

① React+Vite+TypeScript 프로젝트 구성  
② Tailwind CSS와 공통 컴포넌트 설정  
③ Header, Footer, Layout  
④ 메인 페이지  
⑤ 기관 소개  
⑥ 사업 목록·상세  
⑦ 반응형 메뉴  
⑧ API 연결 전 Mock 데이터 적용

**완료 기준**

- 주요 공개 페이지 탐색 가능
- 모바일·데스크톱 레이아웃 정상
- 컴포넌트 중복 최소화

### 3.4 4주차: 백엔드와 관리자

① Express+TypeScript 서버 구성  
② MongoDB Atlas 연결  
③ 데이터 모델과 인덱스  
④ 공개 조회 API  
⑤ Firebase 관리자 인증  
⑥ 관리자 콘텐츠 CRUD  
⑦ 파일 업로드  
⑧ 기관 기본정보 공통 관리

**완료 기준**

- 관리자가 공지와 교육을 등록하고 공개 화면에서 확인
- 권한 없는 사용자의 관리자 API 접근 차단
- 이미지와 첨부파일 저장 확인

### 3.5 5주차: 신청·문의·검색

⑥ 통합 검색  
⑦ 관련 콘텐츠 연결

**완료 기준**

- 신청 제출부터 관리자 확인까지 전체 흐름 통과
- 문의 접수와 알림 정상

### 3.6 6주차: 검수와 배포

① 초기 콘텐츠 입력

⑨ 관리자 매뉴얼 작성

## 6. 바이브코딩 작업 단위

AI 코딩 도구에 전체 홈페이지를 한 번에 요청하지 않고 다음 단위로 나눈다.

① 프로젝트 초기 설정  
② 공통 레이아웃  
③ 메인 섹션 하나씩  
④ 공통 게시판  
⑤ 사업 상세  
⑥ 교육 목록·상세  
⑦ 신청 폼  
⑧ 관리자 인증  
⑨ 콘텐츠 CRUD  
⑩ 검색  
⑪ 테스트와 배포

각 단위마다 다음을 확인한다.

- 요구사항 충족
- 기존 기능 회귀 오류
- 모바일 화면
- 오류 처리
- 보안
- 커밋 기록

## 7. Git 브랜치와 커밋 운영

```text
main           운영 배포
└── develop    통합 개발
    ├── feature/layout
    ├── feature/content-api
    ├── feature/program-apply
    └── feature/admin
```

1인 개발에서는 `main + feature` 방식으로 단순화할 수 있다. 기능이 완료될 때마다 작은 단위로 커밋하고, AI가 만든 대량 변경을 검토하지 않은 채 한 번에 병합하지 않는다.

## 8. 검수 체크포인트

### 기획 검수

- 기관 성격과 메뉴가 일치하는가
- 협회형 회원사 구조가 불필요하게 남아 있지 않은가
- 실제 운영 가능한 게시판 수인가

### 기능 검수

- 신청과 문의가 실제로 저장되는가
- 관리자 권한이 제대로 제한되는가
- 첨부파일이 안전하게 처리되는가

### 콘텐츠 검수

- 기관명, 주소, 연락처가 모든 화면에서 같은가
- 외부자료 출처가 표시되는가
- 검증되지 않은 실적이나 표현이 없는가

### 배포 검수

- Vercel 프런트에서 Render API 호출 가능
- CORS 오류 없음
- 환경변수 노출 없음
- 새로고침 시 SPA 경로 오류 없음
- 모바일과 주요 브라우저 정상

## 9. 일정 위험과 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| 기관 콘텐츠 미확정 | 화면 완성 지연 | 임시문구와 확정문구 구분, 3주차까지 확정 |
| 메뉴 과다 | 개발·운영 부담 | MVP 메뉴 우선, 빈 메뉴 비노출 |
| AI 코드 오류 | 기능 불안정 | 작은 단위 생성, 테스트 후 병합 |
| 개인정보 설계 부족 | 법적 위험 | 최소 수집, 동의문 검토, 접근권한 제한 |
| 무료 배포 지연 | 초기 응답 느림 | 유료 플랜 검토, 상태 페이지와 헬스체크 |
| 파일 저장 문제 | 자료 유실 | 객체 저장소와 백업 사용 |
| 외부자료 저작권 | 게시 중단 위험 | 요약·출처·원문 링크 중심 운영 |

## 10. 1차 공개 후 4주 운영 안정화

- 1주차: 오류와 사용자 문의 집중 대응
- 2주차: 검색어와 이탈 화면 분석
- 3주차: 콘텐츠 분류와 메인 노출 조정
- 4주차: P1 기능 우선순위 재결정

초기 공개를 끝으로 보지 않고 실제 운영 데이터를 바탕으로 메뉴와 기능을 정리한다.
